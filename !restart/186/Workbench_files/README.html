Laboratório - Infraestrutura de monitoramento

Visão geral do laboratório

A capacidade de monitorar aplicações e infraestrutura é essencial para o fornecimento de serviços de TI confiáveis e consistentes.

Os requisitos de monitoramento variam da coleta de estatísticas para análise de longo prazo até a reação rápida a alterações e interrupções. O monitoramento também pode oferecer suporte a relatórios de conformidade, verificando continuamente se a infraestrutura está atendendo aos padrões organizacionais.

Este laboratório mostra como usar as métricas do Amazon CloudWatch, Amazon CloudWatch Logs, Amazon CloudWatch Events e AWS Config para monitorar suas aplicações e infraestrutura.

O laboratório demonstrará como:

  • Usar o Comando de execução do AWS Systems Manager para instalar o agente do CloudWatch em instâncias do Amazon EC2
  • Monitorar logs de aplicações usando o CloudWatch Agent e o CloudWatch Logs
  • Monitorar métricas do sistema usando o CloudWatch Agent e as métricas do CloudWatch
  • Criar notificações em tempo real usando o CloudWatch Events
  • Acompanhar a conformidade da infraestrutura usando o AWS Config

Duração

Este laboratório levará aproximadamente 40 minutos para ser concluído.

Tarefa 1: Instalar o agente do CloudWatch

O agente do CloudWatch pode ser usado para coletar métricas de instâncias do Amazon EC2 e servidores locais, incluindo:

  • Métricas no nível do sistema de instâncias do Amazon EC2, como alocação de CPU, espaço livre em disco e utilização de memória. Essas métricas são coletadas da própria máquina e complementam as métricas padrão do Amazon CloudWatch coletadas pelo CloudWatch.
  • Métricas no nível do sistema de servidores locais, permitindo o monitoramento de ambientes híbridos e servidores não gerenciados pela AWS.
  • Logs de sistema e aplicação de servidores Linux e Windows.
  • Métricas personalizadas de aplicações e serviços usando os protocolos StatsD e collectd.

Nesta tarefa, você usará o AWS Systems Manager para instalar o CloudWatch Agent em uma instância do Amazon EC2. Você o configurará para coletar métricas de aplicação e sistema.

  1. No Console de Gerenciamento da AWS, no menu Services (Serviços), clique em Systems Manager.

  2. No painel de navegação esquerdo, clique em Run Command (Execução de comandos).

    Se não houver um painel de navegação visível, clique no ícone no canto superior esquerdo para fazer com que ele apareça.

    Você usará o Run Command (Execução de comandos) para implantar um comando pré-escrito que instala o CloudWatch Agent.

  3. Clique em Run a Command (Execução de comandos)

  4. Selecione o botão de opção ao lado de AWS-ConfigureAWSPackage (normalmente aparece na parte superior da lista).

  5. Role para baixo até a seção Command parameters (Parâmetros de comando).

    • Action (Ação): Install (Instalar)
    • Name (Nome): AmazonCloudWatchAgent
    • Version (Versão): latest (mais recente)

  6. Na seção Targets (Destinos), selecione Choose instances manually (Escolher instâncias manualmente) e, em seguida, selecione Web Server (Servidor Web).

Essa configuração instalará o CloudWatch Agent no servidor Web.

  1. Clique em Run (Executar) na parte inferior da página.
  2. Aguarde até que o Overall status (Status geral) seja alterado para Success (Com êxito). Ocasionalmente, você pode clicar em atualizar no topo da página para atualizar o status.

Você pode visualizar a saída do trabalho para confirmar que ele foi executado com êxito.

  1. Em Targets and outputs (Destinos e saídas), clique no nome da instância exibido em Instance ID (ID da instância)
  2. Expanda Step 1 - Output (Etapa 1 - Saída).

Você deverá ver a mensagem: Successfully installed arn:aws:ssm:::package/AmazonCloudWatchAgent (Arn instalado com sucesso: aws:ssm:::package/AmazonCloudWatchAgent)

Tarefa 2: Monitorar logs de aplicação usando o CloudWatch Logs

Você pode usar o Amazon CloudWatch Logs para monitorar aplicações e sistemas usando os dados de log. Por exemplo, o CloudWatch Logs pode rastrear o número de erros que ocorrem nos logs da aplicação e enviar a você uma notificação sempre que a taxa de erros exceder um limite especificado.

O CloudWatch Logs usa seus dados de log existentes para fazer o monitoramento; ou seja, as alterações de código não são necessárias. Por exemplo, você pode monitorar os logs de aplicações de termos literais específicos (como "NullReferenceException") ou contar o número de ocorrências de um termo literal em uma posição específica nos dados de log (como códigos de status "404" no log de acesso do servidor Web). Quando o termo que você está pesquisando é encontrado, o CloudWatch Logs reporta os dados para a métrica do CloudWatch que você especificou. Os dados de log são criptografados enquanto estão em trânsito e durante o repouso.

Nesta tarefa, você gerará dados de log no servidor web e, em seguida, monitorará os logs usando o CloudWatch Logs.

O servidor Web gera dois tipos de dados de log:

  • Logs de acesso
  • Logs de erro

Você começará acessando o servidor web.

  1. Clique no menu suspenso Details (Detalhes) acima das instruções que você está lendo no momento e clique em Show (Exibir). Copie o valor WebServerIP.
  2. Abra uma nova guia do navegador da Web, cole o WebServerIP que você copiou e pressione Enter.

Você verá a Test Page (Página de teste) do servidor web.

Agora, você gerará dados de log tentando acessar uma página que não existe.

  1. Insira /start no URL do navegador e pressione Enter.

Você receberá uma mensagem de erro porque a página não foi encontrada. Isso é aceitável. Ele gerará dados nos logs de acesso que estão sendo enviados para o CloudWatch Logs.

  1. Mantenha essa guia aberta no navegador da web, mas volte para a guia do navegador que mostra o Console de Gerenciamento da AWS.
  2. No menu Services (Serviços), escolha CloudWatch.
  3. No painel de navegação esquerdo, clique em Log groups (Grupos de log).

Você deverá ver dois logs listados: HttpAccessLog e HttpErrorLog.

Se esses logs não estiverem listados, espere um minuto e, em seguida, clique em Refresh (Atualizar).

  1. Clique em HttpAccessLog (clique no nome real).
  2. Clique no valor exibido em Logs Streams (Fluxos de logs).

Os dados de log devem ser exibidos, consistindo em solicitações GET que foram enviadas ao servidor web. Você pode visualizar mais informações expandindo as linhas. Os dados de log incluem informações sobre o computador e o navegador que fez a solicitação.

Você deverá ver uma linha com sua solicitação /start com um código de 404, o que significa que a página não foi encontrada.

Isso demonstra como os arquivos de log podem ser enviados automaticamente de uma instância do Amazon EC2 ou de um servidor local para o CloudWatch Logs. Os dados de log podem ser acessados sem a necessidade de fazer login em cada servidor individual. Os dados de log também podem ser coletados de vários servidores, como uma frota de Auto Scaling de servidores web.

 

Criar um filtro de métrica no CloudWatch Logs

Agora você configurará um filtro para identificar Erros 404 no arquivo de log. Isso normalmente seria uma indicação de que o servidor web está gerando links inválidos nos quais os usuários estão clicando.

  1. No painel de navegação esquerdo, clique em Log groups (Grupos de log).
  2. Selecione HttpAccessLog (clique no círculo, não no link).
  3. Em Actions (Ações), selecione Create metric filter (Criar filtro de métrica).

Um filter pattern (padrão de filtro) define os campos no arquivo de log e filtra os dados para valores específicos.

  1. Cole essa linha em Filter Pattern (Padrão de filtro):

    xxxxxxxxxx
    [ip, id, user, timestamp, request, status_code=404, size]

    Isso informa ao CloudWatch Logs como interpretar os campos nos dados de log e define um filtro para encontrar apenas linhas com status_code=404, que indica que uma página não foi encontrada.

  2. Em Test pattern (Padrão de teste), use o menu suspenso para selecionar o id da instância do EC2. Será semelhante ao i-0f07ab62aae4xxxx9.

  3. Selecione o Test pattern (Padrão de teste)

  4. Na seção Results (Resultados) clique em Show test results (Exibir resultados de teste).

Você deverá ver pelo menos um resultado com $status_code de 404. Isso indica que uma página solicitada não foi encontrada.

  1. Escolha Next (Avançar)
  2. Para Filter name (Nome do filtro) insira 404Errors (Erros 404)
  3. Configure estes Metric Details (Detalhes da métrica):
  • Metric Namespace (Namespace de métrica): LogMetrics
  • Metric Name (Nome da métrica): 404Errors (Erros 404)
  • Metric value (Valor da métrica): 1
  1. Escolha Next (Avançar)
  2. Na página Review and create (Revisar e criar), selecione Create File System (Criar sistema de arquivos).

Esse filtro de métrica agora pode ser usado em um alarme.

 

Criar um alarme usando o filtro

Agora, você configurará um alarme para enviar uma notificação quando muitos erros 404 Not Found (404 Não encontrado) forem recebidos.

  1. No painel com título 404Errors (Erros 404), selecione a caixa de seleção no canto superior direito.
  2. Ao lado de Metric filters (Filtros de métrica), selecione Create alarm (Criar alarme)
  3. Defina estas configurações:

  • Period (Período): 1 minute (1 minuto)

  • Conditions (Condições):

    • Whenever 404Errors is: (Sempre que Erros 404 forem:) Greater/Equal (Maior/Igual)
    • que/a): 5

  • Clique em Next (Avançar)

  1. Em Notification (Notificação), configure:
  • Select an SNS Topic (Selecionar um tópico do SNS): Criar novo tópico
  • Email endpoints that will receive the notification (Endpoints de e-mail que receberão a notificação): insira um endereço de e-mail que você possa acessar na sala de aula
  • Clique em Create topic (Criar tópico)
  • Clique em Next (Avançar)
  1. Em Name and description (Nome e descrição), configure:
  • Alarm name (Nome do alarme): 404 Errors (Erros 404)
  • Alarm description (Descrição do alarme): Alert when too many 404s detected on an instance (Alerta quando muitas 404s são detectadas em uma instância)
  • Clique em Next (Avançar)
  1. Clique em Create alarm (Criar alarme)
  2. Acesse seu e-mail, procure uma mensagem de confirmação e selecione o link Confirm subscription (Confirmar assinatura).
  3. Retorne ao Console de Gerenciamento da AWS.
  4. No painel de navegação esquerdo, clique em CloudWatch (na parte superior).

O alarme deverá aparecer em laranja, indicando que há DADOS INSUFICIENTES para acionar o alarme. Isso ocorre porque nenhum dado foi recebido no último minuto.

Agora, você acessará o servidor web para gerar dados de log.

  1. Volte para a guia do navegador com o servidor web.

Se a guia não estiver mais aberta, copie o WebServerIP mostrado à esquerda das instruções que você está lendo no momento e abra uma nova página da web com esse endereço IP.

  1. Tente acessar páginas que não existem adicionando um nome de página após o endereço IP. Repita isso pelo menos 5 vezes.

Por exemplo: http://54.11.22.33/start2

Cada solicitação separada gerará uma entrada de log separada.

  1. Aguarde de 1 a 2 minutos para que o alarme seja acionado. Você pode ocasionalmente clicar em Refresh (Atualizar) para atualizar o status.

O gráfico mostrado na página do CloudWatch deve ficar vermelho para indicar que ele está no estado ALARM (ALARME).

  1. Verifique seu e-mail. Você deve ter recebido um e-mail chamado ALARM: "404 Errors" (ALARME: Erros 404).

Isso demonstra como você pode criar um alarme com base nos dados de log da aplicação e receber alertas quando um comportamento incomum é detectado no arquivo de log. O arquivo de log é facilmente acessível no Amazon CloudWatch Logs para executar análises adicionais para diagnosticar as atividades que levaram o alarme a ser acionado.

 

Tarefa 3: Monitorar métricas de instância usando o CloudWatch

As métricas são dados sobre o desempenho de seus sistemas. O Amazon CloudWatch armazena métricas para serviços da AWS que você usa. Você também pode publicar suas próprias métricas de aplicações por meio do CloudWatch Agent ou diretamente da sua aplicação. O Amazon CloudWatch pode apresentar as métricas para pesquisa, gráficos, painéis e alarmes.

Na tarefa, você usará as métricas de exploração fornecidas pelo Amazon CloudWatch.

  1. No menu Services (Serviços), clique em EC2.
  2. No painel de navegação esquerdo, clique em Instances (Instâncias).
  3. Selecione Web Server (Servidor Web).
  4. Clique no guia Monitoring (Monitoramento) na parte inferior da página.

Examine as métricas apresentadas. Você também pode clicar em um gráfico para exibir mais informações.

O CloudWatch captura métricas sobre o uso de CPU, disco e rede na instância. Essas métricas visualizam a instância "de fora" como uma máquina virtual, mas não fornecem informações sobre o que está sendo executado "dentro" da instância, como, por exemplo, medir a memória livre ou o espaço livre em disco. Felizmente, você pode obter informações sobre o que está acontecendo dentro da instância usando informações capturadas pelo CloudWatch Agent, pois o CloudWatch Agent é executado dentro da instância para coletar métricas.

  1. No menu Services (Serviços), selecione CloudWatch.
  2. No painel de navegação à esquerda, selecione Metrics (Métricas).

A metade inferior da página exibirá as várias métricas coletadas pelo CloudWatch. Algumas são geradas automaticamente pela AWS, enquanto outras foram coletadas pelo CloudWatch Agent.

  1. Clique em CWAgent e, em seguida, em device, fstype, host, path (dispositivo, fstype, host, caminho).

Você verá as métricas de espaço em disco que estão sendo capturadas pelo CloudWatch Agent.

  1. Clique em CWAgent (na linha que diz All > CWAgent > device, fstype, host, path (Todos > CWAgent > dispositivo, fstype, host, caminho)).
  2. Clique em host.

Você verá métricas relacionadas à memória do sistema.

  1. Clique em All (Todos) (na linha que diz All > CWAgent > device, fstype, host, path (Todos > CWAgent > dispositivo, fstype, host, caminho)).

Explore as outras métricas que estão sendo capturadas pelo CloudWatch. Essas são métricas geradas automaticamente provenientes dos serviços da AWS que foram usadas nessa conta da AWS.

Você pode selecionar métricas que deseja que apareçam no gráfico.

 

Tarefa 4: Criar notificações em tempo real

O Amazon CloudWatch Events oferece um fluxo quase em tempo real de eventos do sistema que descreve alterações nos recursos da AWS. Regras simples podem combinar eventos e roteá-los para um ou mais fluxos ou funções de destino. O CloudWatch Events torna-se ciente das alterações operacionais à medida que elas ocorrem.

O CloudWatch Events responde a essas alterações operacionais e realiza ações corretivas, conforme necessário, enviando mensagens para responder ao ambiente, ativando funções, fazendo alterações e capturando informações de estado. Você também pode usar o CloudWatch Events para programar ações automatizadas que são acionadas automaticamente em determinados momentos usando expressões cron ou rate.

Nesta tarefa, você criará uma notificação em tempo real que informará quando uma instância for interrompida ou encerrada.

  1. No painel de navegação esquerdo, clique em Rules (Regras).
  2. Clique em Create rule (Criar regra)
  3. Na seção Event Source (Origem do evento), configure:
  • Service Name (Nome do serviço): EC2
  • Event Type (Tipo de evento): EC2 Instance State-change Notification (Notificação de alteração do estado da instância do EC2)
  • Estado(s) específico(s)
  • No menu suspenso, selecione stopped (interrompido) e terminated (encerrado)
  1. Na seção Targets (Destinos) à direita, configure:
  • Add target (Adicionar destino)
  • Clique em Lambda function (Função do Lambda) e altere-a para SNS topic (Tópico do SNS)
  • Topic (Tópico): selecione a opção Default_CloudWatch_Alarms_Topic.
  1. Clique em Configure details (Configurar detalhes) (na parte inferior da página).
  2. Em Rule definition (Definição de regra), configure:
  • Name (Nome): Instance_Stopped_Terminated
  • Clique em Create rule (Criar regra)

 

Configurar notificação em tempo real

Além de receber um e-mail, você pode configurar o Amazon Simple Notification Service (SNS) para enviar uma notificação ao seu telefone, via SMS.

  1. No menu Services (Serviços), clique em Simple Notification Service (Serviço de notificação simples).
  2. No painel de navegação esquerdo, clique em Topics (Tópicos).
  3. Clique no link na coluna Name (Nome).

Você deve ver uma única assinatura associada ao seu endereço de e-mail. Agora, você adicionará uma notificação SMS.

Se não tiver um telefone que possa receber mensagens SMS, ignore esta etapa.

  1. Clique em Create subscription (Criar assinatura), depois configure:
  • Protocol (Protocolo): SMS
  • Endpoint: insira seu número de telefone celular no formato internacional (por exemplo, +14155557000 ou +917513200000)
  • Clique em Create subscription (Criar assinatura)

Agora você está pronto para acionar um alerta em tempo real.

  1. No menu Services (Serviços), clique em EC2.
  2. No painel de navegação esquerdo, clique em Instances (Instâncias).
  3. Selecione Web Server (Servidor Web).
  4. Clique em Instance state (Estado da instância), em Stop instance (Parar instância) e em Stop (Parar)

A instância Web Server (servidor Web) entrará no estado de stopping (interrupção). Depois de um minuto, ela entrará no estado stopped (interrompido).

Em seguida, você deve receber uma mensagem SMS com detalhes da instância que foi interrompida.

A mensagem é formatada em JSON. Para receber uma mensagem mais fácil de ler, você pode criar uma função do AWS Lambda que é acionada pelo CloudWatch Events. A função do Lambda pode formatar uma mensagem mais amigável e enviá-la por meio do Amazon SNS.

Isso demonstra como é fácil receber notificações em tempo real quando a infraestrutura muda.

 

Tarefa 5: Monitorar a conformidade da infraestrutura

O AWS Config é um serviço que permite avaliar, auditar e examinar as configurações de seus recursos de AWS. O config monitora e registra continuamente suas configurações de recursos da AWS e permite automatizar a avaliação das configurações registradas em relação às configurações desejadas.

Com o Config, você pode analisar alterações feitas nas configurações e relacionamentos entre os recursos da AWS, aprofundar-se de forma detalhada no histórico de configuração de recursos e determinar a conformidade geral em relação às configurações especificadas em suas diretrizes internas. Isso permite simplificar a auditoria de conformidade, a análise de segurança, o gerenciamento de alterações e a solução de problemas operacionais.

Nesta tarefa, você ativará o AWS Config Rules para garantir a conformidade da marcação e dos volumes do EBS.

  1. No menu Services (Serviços), clique em Config.
  2. Se for exibido o botão Get started (Comece a usar), faça o seguinte:
  • Clique em Get started (Comece a usar)
  • Clique em Next (Avançar). Se você receber um erro, em AWS Config role (Função do AWS Config) na parte inferior da página, selecione Use an existing AWS Config service-linked role (Use uma função vinculada ao serviço existente do AWS Config).
  • Clique em Skip (Ignorar)
  • Clique em Confirm (Confirmar)

Isso configurará o AWS Config para uso inicial.

  1. No painel de navegação esquerdo, clique em Rules (Regras) (aquela na parte superior).
  2. Clique em Add rule (Adicionar regra)
  3. No campo de pesquisa, insira: required-tags
  4. Clique na caixa required-tags exibida.

Você configurará a regra para exigir um código de projeto para cada recurso.

  1. Role para baixo até Rule parameters (Parâmetros da regra) e configure:
  • À direita de tag1Key, insira: project (projeto) (substitua qualquer valor existente)
  • Clique em Save (Salvar) (na parte inferior da página)

Essa regra agora procurará recursos que não têm uma tag project (projeto). Isso levará alguns minutos para ser concluído, portanto, continue com as próximas etapas. Não há necessidade de esperar.

Agora você adicionará uma regra que procura volumes do Amazon EBS que não estão anexados às instâncias do Amazon EC2.

  1. Clique em Add rule (Adicionar regra)
  2. No campo de pesquisa, insira: ec2-volume-inuse-check
  3. Clique na caixa ec2-volume-inuse-check que for exibida.
  4. Clique em Save (Salvar)
  5. Aguarde até que pelo menos uma das regras tenha concluído a avaliação. Clique em Refresh (Atualizar) no canto superior direito a cada 60 segundos para atualizar o status.

Se você receber uma mensagem informando que não há recursos no escopo, aguarde alguns minutos mais. Essa mensagem é uma indicação de que o AWS Config ainda está verificando os recursos disponíveis. A mensagem acabará desaparecendo.

  1. Clique em cada uma das regras para visualizar o resultado das auditorias.

Entre os resultados devem estar:

  • required-tags: uma instância do EC2 compatível (porque o servidor web tem uma tag project (projeto)) e muitos recursos não compatíveis que não têm uma tag project (projeto)
  • ec2-volume-inuse-check: um volume compatível (anexado a uma instância) e um volume não compatível (não anexado a uma instância)

O AWS Config tem uma grande biblioteca de verificações de conformidade predefinidas e você pode criar verificações adicionais escrevendo sua própria regra do AWS Config usando o AWS Lambda.

Um painel será exibido indicando que "DELETE has been initiated... You may close this message box now." (A EXCLUSÃO foi iniciada... Você pode fechar esta caixa de mensagem agora).
