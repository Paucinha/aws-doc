Atividade - Migrar para o Amazon RDS

Visão geral da atividade

Nesta atividade, você migrará a aplicação Web do Café para usar uma instância totalmente gerenciada do Amazon Relational Database Service (Amazon RDS), em vez de uma instância de banco de dados local.

Este diagrama ilustra a topologia do ambiente de tempo de execução da aplicação Web do Café antes da migração. O banco de dados da aplicação é executado em uma instância do Amazon Elastic Compute Cloud (Amazon EC2) LAMP (Linux, servidor Web Apache, MySQL e PHP) com o código da aplicação. A instância tem um tipo de t3 pequeno e é executada em uma sub-rede pública para permitir que clientes da Internet acessem o site. Uma instância CLI Host (Host da CLI) reside na mesma sub-rede para permitir a administração da instância usando a Interface de linhas de comando da AWS (CLI da AWS).

Diagrama da topologia do banco de dados antes da migração

O diagrama a seguir ilustra a topologia do ambiente de tempo de execução da aplicação Web do Café após a migração.

Você migrará o banco de dados local do Café para um banco de dados do Amazon Relational Database Service (Amazon RDS) que está fora da instância. O banco de dados do RDS será implantado na mesma nuvem privada virtual (VPC) da instância.

Durante o processo de migração, você criará os componentes ilustrados neste diagrama, incluindo duas sub-redes privadas em diferentes zonas de disponibilidade, um grupo de segurança para a instância de banco de dados e a própria instância de banco de dados do RDS. Após a migração do banco de dados, você vai reconfigurar a aplicação do Café para usar o banco de dados da instância do RDS, em vez do banco de dados local.

Objetivos da atividade

Depois de concluir esta atividade, você poderá:

  • Criar uma instância MariaDB do Amazon RDS usando a AWS CLI.
  • Migrar dados de um banco de dados MariaDB local para um banco de dados MariaDB do Amazon RDS.
  • Monitorar uma instância do Amazon RDS usando as métricas do Amazon CloudWatch.

 

Relevância de casos de negócio

Um novo requisito de negócio para o Café: automatizar tarefas de manutenção de banco de dados e reduzir custos de administração de banco de dados

A administração do banco de dados do Café está ficando difícil para a Sofía e Nikhil. Por exemplo, fazer backups semanais do banco de dados exige que eles trabalhem horas extras nos fins de semana porque precisam enviar manualmente o trabalho de backup. Eles também devem verificar sua execução bem-sucedida durante as horas em que a empresa está fechada.

Além disso, eles tiveram dificuldades recentemente com a instalação de um patch necessário e mal conseguiram completá-lo durante o fim de semana após o horário. Martha e Frank também estão percebendo que essas tarefas de manutenção estão aumentando os custos trabalhistas da empresa, pois eles precisam pagar pelas horas extras que Sofia e Nikhil devem trabalhar. Assim, eles entraram em contato com a equipe da AWS para solicitar algumas recomendações.

Olivia, arquiteta de soluções da AWS, analisou a solicitação de Martha e Frank e sugeriu que migrassem o banco de dados para o Amazon RDS. Essa solução reduzirá a carga manual das tarefas comuns de manutenção do banco de dados, como backups, instalação de patches e upgrades. Como um serviço totalmente gerenciado, o Amazon RDS executa essas tarefas automaticamente.

Nesta atividade, você assumirá os papéis de Nikhil e Sofía, e trabalhará na migração do banco de dados do Café para o Amazon RDS.

Tarefa 1.3: Criar componentes de pré-requisito

Nesta tarefa, você cria os componentes de infraestrutura de pré-requisitos para a instância do Amazon RDS. Especificamente, você cria os seguintes componentes que são mostrados no Diagrama da topologia após a migração

  • CafeDataBaseSG (Grupo de segurança)
  • Sub-rede privada 1 do CafeDB
  • Sub-rede privada 2 do CafeDB
  • Grupo de sub-redes do CafeDB (Grupo de sub-redes de banco de dados)

  1. Crie o grupo de segurança CafeDatabaseSG. Esse grupo de segurança é usado para proteger a instância do Amazon RDS. Ele deverá ter uma regra de entrada que permita somente solicitações do MySQL (usando o protocolo TCP padrão e a porta 3306) das instâncias associadas ao CafeSecurityGroup. Essa regra garante que somente CafeInstance seja capaz de acessar o banco de dados. Na janela SSH, insira:

    xxxxxxxxxx
    aws ec2 create-security-group \
    --group-name CafeDatabaseSG \
    --description "Security group for Cafe database" \
    --vpc-id <CafeInstance VPC ID>

    No comando, substitua <CafeInstance VPC ID> pelo valor que você gravou anteriormente.

  2. Depois que o comando for concluído, anote o GroupID retornado como:

    xxxxxxxxxx
    CafeDatabaseSG Group ID: sg-zzzzzzzzzz

  3. Em seguida, crie a regra de entrada para o grupo de segurança. Na janela SSH, insira:

    xxxxxxxxxx
    aws ec2 authorize-security-group-ingress \
    --group-id <CafeDatabaseSG Group ID> \
    --protocol tcp --port 3306 \
    --source-group <CafeSecurityGroup Group ID>

    No comando, substitua <CafeDatabaseSG Group ID> e *<CafeSecurityGroup Group ID>** pelos valores que você anotou anteriormente. Tenha cuidado para fazer as substituições corretas.

  4. Execute o seguinte comando para confirmar se a regra de entrada foi aplicada adequadamente:

    xxxxxxxxxx
    aws ec2 describe-security-groups \
    --query "SecurityGroups[*].[GroupName,GroupId,IpPermissions]" \
    --filters "Name=group-name,Values='CafeDatabaseSG'"

    A saída do comando deve mostrar que o grupo de segurança CafeDatabaseSG agora tem uma regra de entrada que permite conexões da porta TCP 3306, se a origem da conexão for uma instância que tenha a associação do ID do grupo CafeSecurityGroup.

    Ao criar uma instância de banco de dados do Amazon RDS em uma VPC, você deve ter duas sub-redes privadas ou duas sub-redes públicas disponíveis para que possa formar um grupo de sub-redes de banco de dados. Portanto, você cria duas sub-redes privadas e um grupo de sub-redes de banco de dados em seguida.

  5. Crie a sub-rede privada do CafeDB 1. Essa sub-rede hospeda a instância de banco de dados do Amazon RDS. É uma sub-rede privada definida na mesma zona de disponibilidade do CafeInstance.

Atribua à sub-rede um bloco de endereço de Roteamento entre Domínios sem Classificação (CIDR) que esteja dentro do intervalo de endereços da VPC, mas que não substitua o intervalo de endereços de qualquer outra sub-rede na VPC. Este é o motivo pelo qual você coletou as informações sobre a VPC e os blocos CIDR de sub-rede existentes:

  • Bloco CIDR IPv4 da VPC do Café: 10.200.0.0/20
  • Bloco CIDR IPv4 da sub-rede pública 1 do Café: 10.200.0.0/24

Considere esses intervalos de endereços. Você consegue encontrar um bloco CIDR adequado para a sub-rede privada? Uma resposta possível seria usar o intervalo de endereços 10.200.2.0/23.

  1. Na janela SSH, insira:

    xxxxxxxxxx
    aws ec2 create-subnet \
    --vpc-id <CafeInstance VPC ID> \
    --cidr-block 10.200.2.0/23 \
    --availability-zone <CafeInstance Availability Zone>

    No comando, substitua <CafeInstance VPC ID> e <CafeInstance Availability Zone> pelos valores que você registrou anteriormente.

  2. Depois que o comando for concluído, grave o SubnetId retornado como:

    xxxxxxxxxx
    Cafe Private Subnet 1 ID: subnet-ssssssssss

  3. Crie a sub-rede privada do CafeDB 2. Esta é a sub-rede extra necessária para formar o grupo de sub-redes do banco de dados. É uma sub-rede privada vazia definida em uma zona de disponibilidade diferente da CafeInstance.

De forma semelhante ao que você fez na etapa anterior, atribua um bloco de endereços CIDR à sub-rede que esteja dentro do intervalo de endereços da VPC, mas que não substitua o intervalo de endereços de qualquer outra sub-rede da VPC. Até agora, você usou estes intervalos de endereço:

  • Bloco CIDR IPv4 da VPC do Café: 10.200.0.0/20
  • Bloco CIDR IPv4 da sub-rede pública 1 do Café: 10.200.0.0/24
  • Bloco CIDR IPv4 da Sub-rede privada 1 do Café: 10.200.2.0/23

Como sugestão, use o intervalo de endereços 10.200.10.0/23 para essa segunda sub-rede privada.

Além disso, lembre-se de uma etapa anterior da lista de zonas de disponibilidade disponíveis na região. Você determinou que o CafeDB está na primeira zona de disponibilidade (por exemplo, se a região for us-west-2, estará em us-west-2a). Portanto, escolha qualquer outra zona de disponibilidade (mas não a que termina com a) para a zona de disponibilidade da segunda sub-rede.

  1. Na janela SSH, insira:

    xxxxxxxxxx
    aws ec2 create-subnet \
    --vpc-id <CafeInstance VPC ID> \
    --cidr-block 10.200.10.0/23 \
    --availability-zone <availability-zone>

    No comando, substitua <CafeInstance VPC ID> pelo valor que você gravou anteriormente. Além disso, substitua <availability-zone> pela zona de disponibilidade real.

  2. Depois que o comando for concluído, grave o SubnetId retornado como:

    xxxxxxxxxx
    Cafe Private Subnet 2 ID: subnet-tttttttttt

  3. Crie o grupo de sub-rede CafeDB. O grupo de sub-redes de banco de dados (DB) é uma coleção de sub-redes em uma VPC. Ao criar uma instância do Amazon RDS, você especifica a VPC em que a instância do RDS reside, associando a instância a um grupo de sub-redes de banco de dados.

Para a instância do RDS do Café, o grupo de sub-redes de banco de dados consiste nas duas sub-redes privadas que você criou nas etapas anteriores, sub-rede privada 1 do Café e sub-rede privada 2 do Café.

  1. Na janela SSH, insira:

    xxxxxxxxxx
    aws rds create-db-subnet-group \
    --db-subnet-group-name "CafeDB Subnet Group" \
    --db-subnet-group-description "DB subnet group for Cafe" \
    --subnet-ids <Cafe Private Subnet 1 ID> <Cafe Private Subnet 2 ID> \
    --tags "Key=Name,Value= CafeDatabaseSubnetGroup"

    No comando, substitua <Cafe Private Subnet 1 ID> e <Cafe Private Subnet 2 ID> pelos valores que você gravou anteriormente. Observe que existe um espaço entre os IDs de sub-rede.

    Depois que o comando for concluído, ele retornará os atributos do grupo de sub-redes de banco de dados.

 

Tarefa 1.4: Criar a instância MariaDB do Amazon RDS

Agora você pode criar a CafeDBInstance que é mostrada no diagrama da topologia após a migração. Usando a CLI da AWS, crie uma instância MariaDB do Amazon RDS com as seguintes definições de configuração:

  • DB instance identifier (Identificador de instância de banco de dados): CafeDBInstance
  • Engine option (Opção de mecanismo): MariaDB
  • DB engine version (Versão do mecanismo de banco de dados): 10.5.13
  • DB Instance class (Classe da instância): db.t3.micro
  • Allocated storage (Armazenamento alocado): 20 GB
  • Availability Zone (Zona de disponibilidade): Zona de disponibilidade do CafeInstance
  • DB Subnet group (Grupo de sub-rede DB): Grupo de sub-rede do CafeDB
  • VPC security groups (Grupos de segurança da VPC): CafeDataBaseSG
  • Public accessibility (Acessibilidade pública): Não
  • Username (Nome de usuário): root
  • Master password (Senha principal): Re:Start!9

Essas opções especificam a criação de uma instância de banco de dados MariaDB implantada na mesma zona de disponibilidade da instância do Cafe. Ela também usa o grupo de sub-rede de banco de dados criado na etapa anterior.

Dica: use a documentação da AWS CLI para RDS para ajudar a determinar a sintaxe correta dos comandos da AWS CLI que você deverá usar nesta tarefa.

  1. Na janela SSH, insira:

    xxxxxxxxxx
    aws rds create-db-instance \
    --db-instance-identifier CafeDBInstance \
    --engine mariadb \
    --engine-version 10.5.13 \
    --db-instance-class db.t3.micro \
    --allocated-storage 20 \
    --availability-zone <CafeInstance Availability Zone> \
    --db-subnet-group-name "CafeDB Subnet Group" \
    --vpc-security-group-ids <CafeDatabaseSG Group ID> \
    --no-publicly-accessible \
    --master-username root --master-user-password 'Re:Start!9'

    No comando, substitua <CafeInstance Availability Zone> e <CafeDatabaseSG Group ID> pelos valores que você anotou anteriormente.

    Embora o comando retorne algumas informações sobre o banco de dados imediatamente, a instância de banco de dados pode levar até dez minutos para se tornar disponível.

  2. Monitore o status da instância de banco de dados e aguarde até que ele mostre o valor avaiable (disponível). Na janela SSH, insira:

    xxxxxxxxxx
    aws rds describe-db-instances \
    --db-instance-identifier CafeDBInstance \
    --query "DBInstances[*].[Endpoint.Address,AvailabilityZone,PreferredBackupWindow,BackupRetentionPeriod,DBInstanceStatus]"

    Esse comando exibe o status da instância de banco de dados como o último valor retornado. Ele também mostra os seguintes valores do banco de dados:

    • Endereço do endpoint
    • Zona de disponibilidade
    • Janela de backup preferencial
    • Período de retenção de backup

    Os backups automatizados ocorrem diariamente durante a janela de backup preferencial e são retidos pela duração especificada pelo período de retenção de backup. Observe o valor 1 para o período de retenção de backup, que indica que, por padrão, os backups diários são retidos por apenas 1 dia. Além disso, a janela de backup preferencial é definida com um intervalo de 30 minutos por padrão. Você pode modificar essas configurações para corresponder à política de backup desejada.

    O atributo de status mostrará inicialmente um valor creating (criando) e depois mudará para modifying (modificando). Aguarde alguns momentos e repita o comando novamente. O status mudará progressivamente para backing-up (fazendo backup) e finalmente para available (disponível).

  3. Continue repetindo o comando até que o status mostre available (disponível) e, em seguida, anote o valor retornado para Endpoint Address (Endereço de endpoint) usando o seguinte formato:

    xxxxxxxxxx
    RDS Instance Database Endpoint Address: Cafedb.xxxxxxx.us-west-2.rds.amazonaws.com

 

Tarefa 2: Migrar dados de aplicações para a instância do Amazon RDS

Nesta tarefa, você migrará os dados do banco de dados local existente para o banco de dados recém-criado do Amazon RDS. Especificamente, você:

a) Abrirá uma sessão SSH para o CafeInstance.

b) Usará o utilitário mysqldump para criar um backup do banco de dados local.

c) Restaurará o backup para o banco de dados do Amazon RDS.

d) Testará a migração de dados.

Observe que você executará estas etapas na linha de comando de uma sessão SSH conectada ao CafeInstance. Essa instância pode se comunicar com êxito com a instância do Amazon RDS usando o protocolo do MySQL porque você associou o grupo de segurança CafeDatabaseSG à instância do RDS.

  1. Comece abrindo uma sessão SSH no CafeInstance e fazendo login como ec2-user. Siga as mesmas instruções que você usou na Tarefa 1.1, mas certifique-se de usar o endereço IP público do CafeInstance, que você gravou em uma etapa anterior:
  • Endereço IP público do CafeInstance: nnn.nnn.nnn.nnn

  1. Use o utilitário mysqldump para criar um backup do banco de dados local cafe_db. Este programa utilitário faz parte do produto de banco de dados MySQL e está disponível para você usar. Na janela SSH, insira:

    xxxxxxxxxx
    mysqldump --user=root --password='Re:Start!9' \
    --databases cafe_db --add-drop-database > cafedb-backup.sql

    Esse comando gera instruções de linguagem de consulta estruturada (SQL) em um arquivo chamado cafedb-backup.sql, que podem ser executadas para reproduzir o esquema e os dados do banco de dados cafe_db original.

  2. Revise o conteúdo do arquivo de backup. Abra o arquivo cafedb-backup.sql no seu editor de texto preferido. Como alternativa, se você quiser visualizá-lo usando o comando less do Linux, na janela SSH, digite:

    xxxxxxxxxx
    less cafedb-backup.sql

    Dica: Quando você usar o comando less, use as teclas de seta para cima e para baixo para mover uma linha para cima ou uma linha para baixo, respectivamente. Você também pode usar as teclas PAGE UP e PAGE DOWN para navegar por uma página para cima ou para baixo, respectivamente. Para sair do comando, digite q.

    Observe os vários comandos SQL para criar o banco de dados, criar tabelas e índices e preencher o banco de dados com os dados originais.

  3. Em seguida, restaure o backup para o banco de dados do Amazon RDS usando o comando mysql. Especifique o endereço do endpoint da instância do RDS no comando. Na janela SSH, insira:

    xxxxxxxxxx
    mysql --user=root --password='Re:Start!9' \
    --host=<RDS Instance Database Endpoint Address> \
    < cafedb-backup.sql

    No comando, substitua <RDS Instance Database Endpoint Address> pelo valor que você gravou anteriormente.

    Esse comando cria uma conexão mysql com a instância do RDS e executa as instruções SQL no arquivo cafedb-backup.sql.

  4. Por último, verifique se cafe_db foi criado e preenchido com êxito na instância do Amazon RDS. Abra uma sessão interativa mysql na instância e recupere os dados na tabela de produtos do banco de dados cafe_db. Na janela SSH, insira:

    xxxxxxxxxx
    mysql --user=root --password='Re:Start!9' \
    --host=<RDS Instance Database Endpoint Address> \
    mom_pop_db

    No comando, substitua <RDS Instance Database Endpoint Address> pelo valor que você gravou anteriormente.

  5. Em seguida, insira a instrução SQL para recuperar os dados na tabela de produtos:

    xxxxxxxxxx
    select * from product;

    A consulta deve retornar 9 linhas da tabela.

  6. Agora você pode sair da sessão SQL interativa. Na janela SSH, insira:

    xxxxxxxxxx
    exit

    Deixe a janela SSH de CafeInstance aberta para uso posterior.

 

Tarefa 3: Configurar o site para usar a instância do Amazon RDS

Agora você está pronto para configurar o site do Café para usar a instância do Amazon RDS. Esta etapa é simples porque o designer da aplicação seguiu as práticas recomendadas e externalizou as informações de conexão do banco de dados como parâmetros no repositório de parâmetros do AWS Systems Manager. Nesta tarefa, você altera o parâmetro URL do banco de dados da aplicação do Café para apontar para o endereço do endpoint da instância do RDS.

  1. Na guia do navegador do Console de Gerenciamento da AWS, selecione Services > Systems Manager (Serviços > Systems Manager).
  2. No painel de navegação esquerdo, clique em Parameter Store (Repositório de parâmetros).
  3. Na lista Parameters (Parâmetros), clique em /cafe/dbUrl. O valor atual do parâmetro é exibido, com sua descrição e outras informações de metadados.
  4. Clique em Edit (Editar).
  5. Na página Parameter details (Detalhes de parâmetros), substitua o texto da caixa Value (Valor) pelo valor do Endereço de endpoint do banco de dados da instância do RDS que você registrou anteriormente.
  6. Clique em Save changes (Salvar alterações).

O parâmetro dbUrl agora faz referência à instância do banco de dados do Amazon RDS em vez do banco de dados local.

  1. Teste o site para confirmar se ele consegue acessar o novo banco de dados corretamente. Em uma janela do navegador, insira a seguinte URL:

    xxxxxxxxxx
    http://<CafeInstance Public DNS Name>/cafe

    Substitua <CafeInstance Public DNS Name> pelo valor que você gravou anteriormente.

  2. A página inicial do site deve ser carregada corretamente. Selecione a guia Order History (Histórico de pedidos) e observe o número de pedidos no banco de dados. Compare esse número com o Número de pedidos que você anotou antes da migração do banco de dados. Ambos os números devem corresponder.

  3. Sinta-se livre para fazer alguns novos pedidos e verificar o bom funcionamento do site. Feche a guia do navegador quando terminar.

 

Tarefa 4: Monitorar o banco de dados do Amazon RDS

Um dos benefícios do uso do Amazon RDS é a capacidade de monitorar o desempenho de uma instância de banco de dados. O Amazon RDS envia automaticamente métricas para o Amazon CloudWatch a cada minuto para cada banco de dados ativo. Nesta tarefa, você identifica algumas dessas métricas de desempenho e aprende a monitorar uma métrica no console do Amazon RDS.

  1. Na guia do navegador do Console de Gerenciamento da AWS, selecione Services > RDS (Serviços > RDS).
  2. No painel de navegação esquerdo, clique em Databases (Bancos de dados).
  3. Na lista Databases (Bancos de dados), clique em Cafedbinstance. Informações detalhadas sobre o banco de dados são exibidas.
  4. Clique na guia Monitoring (Monitoramento). Por padrão, essa guia exibe várias métricas importantes da instância de banco de dados disponíveis no Amazon CloudWatch. Cada métrica inclui um gráfico que mostra a métrica à medida que ela é monitorada em um intervalo de tempo específico.

A lista de métricas exibidas incluirá o seguinte:

  • CPU Utilization (Utilização da CPU): percentagem de utilização da CPU
  • DB Connections (Conexões de banco de dados): número de conexões de banco de dados em uso.
  • Free Storage Space (Espaço de armazenamento livre): a quantidade de espaço de armazenamento disponível
  • Write IOPS (IOPS de gravação): número médio de operações de E/S de gravação de disco por segundo
  • Read IOPS (IOPS de leitura): número médio de operações de E/S de leitura de disco por segundo
  • Read Throughput (Taxa de transferência de leitura): número médio de bytes lidos do disco por segundo
  • Write Throughput (Taxa de transferência de gravação): número médio de bytes gravados no disco por segundo

Dica: algumas das métricas listadas acima podem aparecer na segunda ou terceira página de gráficos. Clique em 2 ou 3 à direita da caixa de pesquisa do CloudWatch para ver métricas adicionais.

  1. Monitore a métrica Conexões de banco de dados à medida que você cria uma conexão com o banco de dados do CafeInstance.

    Abra uma sessão SQL interativa na instância cafe_db do RDS. Na janela SSH do CafeInstance, informe:

    xxxxxxxxxx
    mysql --user=root --password='Re:Start!9' \
    --host=<RDS Instance Database Endpoint Address> \
    cafe_db

    No comando, substitua <RDS Instance Database Endpoint Address> pelo valor que você gravou anteriormente.

  2. Insira uma instrução SQL para recuperar os dados na tabela de produtos:

    xxxxxxxxxx
    select * from product;

    A consulta deve retornar 9 linhas da tabela.

  3. No console do Amazon RDS, clique no gráfico Conexões de banco de dados para abri-lo em uma exibição maior. O gráfico mostra uma linha que indica que a conexão 1 está em uso. Essa conexão foi estabelecida pela sessão SQL interativa do CafeInstance.

Dica: se o gráfico não mostrar nenhuma conexão em uso, aguarde um minuto (que é o intervalo de amostragem) e clique em Refresh (Atualizar). Agora deve mostrar uma conexão aberta.

  1. Feche a conexão da sessão SQL interativa. Na janela SSH do CafeInstance, informe:

    xxxxxxxxxx
    exit

  2. Aguarde um minuto e, em seguida, no gráfico DB Connections (Conexões de banco de dados) no console do Amazon RDS, clique em Refresh (Atualizar). O gráfico agora mostra que o número de conexões em uso é 0.

Sinta-se livre para explorar os outros gráficos. Para obter mais informações sobre a definição de cada métrica, consulte o capítulo Monitoramento com o Amazon CloudWatch da documentação do Amazon RDS.
